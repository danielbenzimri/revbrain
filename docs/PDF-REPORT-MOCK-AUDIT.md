# PDF Report — Mock & Fallback Audit

**Status:** Draft for CTO / architectural review
**Audience:** CTO, senior engineers reviewing whether the RevBrain CPQ assessment report is safe for customer delivery.
**Scope:** Every code path that feeds a finding into the rendered PDF — scripts, assembler, templates, validators, collectors (spot-check).
**Method:** Direct code reads against `feat/bb3-wave1` + targeted grep for: hardcoded values, `Math.random`, silent `?? 0` / `?? ''` / `?? 'Unknown'` coercions, swallowed catches, fabricated placeholder findings, section-skipping on failure, wall-clock leakage, and section validators that pass on empty data.
**Non-negotiable the audit is measuring against:** _No customer-facing number may be fabricated, randomized, defaulted, or inferred from absence. Absence of data must be surfaced explicitly — either as an error banner, a suppressed section, or an "Insufficient data" marker — but never as a silent zero._

## 2. Definitions

For the purposes of this audit, the following are all classified as "mocks" regardless of how their authors intended them:

| Class                        | Definition                                                                                                                                                                                                                                           | Example                                                                  |
| ---------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------ |
| **Fabrication**              | A numeric value emitted into report data that is not derived from the extraction. Includes `Math.random`, hardcoded constants, hardcoded percentages, and "estimated" defaults.                                                                      | `Math.round(totalActive * 0.98)` for "Family field populated"            |
| **Silent fallback**          | A code path that substitutes a default value when a finding is absent, without recording that the substitution happened. `x?.countValue ?? 0` when `x === undefined` means "the extractor did not produce this finding", but the report renders `0`. | `(quotes90d?.countValue ?? 0) > 0 ? ... : (quotesAll?.countValue ?? 0)`  |
| **Silent exception swallow** | A `try/catch` that replaces a failure with a default without logging.                                                                                                                                                                                | `try { JSON.parse(refs) } catch { refs = []; }`                          |
| **Wall-clock leakage**       | A customer-facing value that depends on `Date.now()` / `new Date()` at rendering time, producing different reports on different days for the same input.                                                                                             | `assessmentPeriod = "Jan 11 – Apr 11 (90 Days)"` computed at render time |
| **Validator blindness**      | A validation rule whose `passed: true` branch is reachable from a silent fallback or from an empty list that should have been populated.                                                                                                             | V11: empty `fieldCompleteness` list → "passed"                           |

All five classes are treated as policy violations for a customer-facing deliverable. The correct behavior in every case is: **surface the absence loudly**, either by failing the assemble step, emitting a visible validation banner, or suppressing the section with an explicit "data unavailable" marker.

## 4. The systemic `?? 0` fallback pattern — the real story

This is the finding that matters for "going forward, fallback must error."

**Count.** A grep for `?? 0\b`, `?? ''`, `?? []`, `?? 'Unknown'`, `?? 'N/A'` in [apps/worker/src/report/assembler.ts](apps/worker/src/report/assembler.ts) returns **105 matches** across the file. The majority are `?? 0` on `countValue` / finding-lookups.

No individual `?? 0` is a "lie" — each one is defensively coded to survive missing data. **The aggregate effect is a systemic policy violation:** the canonical PDF report cannot distinguish, at render time, between "the extractor measured zero of this thing" and "the extractor never produced a finding for this thing at all." Both render identically.

### Finding 2 — sample sites from the ~105 total

The following are representative. The full list can be enumerated with `grep -n "?? 0\b" apps/worker/src/report/assembler.ts`.

**2a. `totalQuotes` at [assembler.ts:487-488](apps/worker/src/report/assembler.ts#L487-L488):**

```ts
const totalQuotes =
  (quotes90d?.countValue ?? 0) > 0 ? quotes90d!.countValue! : (quotesAll?.countValue ?? 0);
```

If neither the "Quotes (90d)" nor the "Quotes (all)" `DataCount` finding exists — which happens if the `usage` collector fails, times out, or is skipped — `totalQuotes` silently becomes `0`. This value is then used as the denominator for every percentage in §7.2 "Conversion by Size," §7.3 "Top Products" (via `topProductDenominator` at line 520), and several others. Those sections render with "0" in every cell and a customer reads them as "this org has zero activity" rather than "this collector didn't run."

**2b. Cascade into denominator — [assembler.ts:519-520](apps/worker/src/report/assembler.ts#L519-L520):**

```ts
const totalTopProductQuoted = topProducts.reduce((sum, p) => sum + (p.countValue ?? 0), 0);
const topProductDenominator = totalQuotes > 0 ? totalQuotes : totalTopProductQuoted;
```

Defensive fallback on a defensive fallback. If `totalQuotes` is a silent zero (per 2a) and `totalTopProductQuoted` is also zero because the finding list is empty, every top-product percentage renders as `NaN%` or `0%` depending on how the division is guarded downstream.

**2c. Usage/conversion metrics from evidence refs — [assembler.ts:970-972](apps/worker/src/report/assembler.ts#L970-L972):**

```ts
percentQuotes:    Number(s.evidenceRefs?.find((r) => r.label === '% of quotes')?.value ?? 0),
percentRevenue:   Number(s.evidenceRefs?.find((r) => r.label === '% of revenue')?.value ?? 0),
conversionRate:   Number(s.evidenceRefs?.find((r) => r.label === 'conversion %')?.value ?? 0),
```

Three percentage fields, each triple-defaulted: `evidenceRefs` may be missing, the specific ref may not be found, its `value` may be null. Any of the three failure modes produces `0`. If the usage collector's conversion-by-size sub-step failed halfway through, the customer sees a table where every row is "0% / 0% / 0%" and reads it as "nothing converts."

**2d. Assembler entry normalization swallowing bad JSON — [assembler.ts:395-407](apps/worker/src/report/assembler.ts#L395-L407):**

```ts
for (const f of findings) {
  const refs = f.evidenceRefs;
  if (typeof refs === 'string') {
    try {
      (f as Record<string, unknown>).evidenceRefs = JSON.parse(refs);
    } catch {
      (f as Record<string, unknown>).evidenceRefs = [];
    }
  } else if (!Array.isArray(refs)) {
    (f as Record<string, unknown>).evidenceRefs = [];
  }
}
```

Two silent defaults in 13 lines:

1. **Line 402:** `JSON.parse` failure → empty array. If the DB-stored `evidenceRefs` string was truncated or corrupted, the entire set of referenced fields for that finding is lost. The finding still contributes to counts (because it's not dropped), but downstream code looking for `evidenceRefs.find((r) => r.label === '...')` will not find anything, cascading into cases like 2c.
2. **Line 404-406:** non-string non-array → empty array. Same effect via a different entry path.

Neither path logs, emits a diagnostic, or touches the validator. The render continues with a silently-damaged finding.

### Failure mode summary

Take the user's pending scenario: the v2.1 report was mostly-correct because the collectors that did run produced real findings. A stricter test case would be: run the canonical `generate-report.ts` against a partial `assessment-results.json` where **only the discovery and catalog collectors ran** (the other 10 were cancelled or timed out). With 105 silent defaults, the resulting PDF would render:

- §7 (Configuration domain) as if every rule count were really zero
- §7.2 (Conversion by size) as a fully populated table of zeros
- §8 (Approvals) as "no approvals configured"
- §9 (Customizations) as "no formula fields, no validation rules"
- §10 (Related functionality) as "no integrations"

None of those sections would contain a visible error banner unless the validation layer happened to trip one of its cross-consistency checks (V17–V24). The report would look complete and confidently wrong — the exact failure mode the coverage-gaps doc principle P1 warns against.

**This is the structural defect the user is asking about.** Individually each `?? 0` is defensive; systemically they form a trust problem.

## 5. Other defects in the canonical path

### Finding 3 — Wall-clock leakage into `assessmentPeriod`

[assembler.ts:503-509](apps/worker/src/report/assembler.ts#L503-L509):

```ts
// Assessment period calculation
const assessmentDate = new Date().toISOString().split('T')[0];
const endDate = new Date();
const startDate = new Date(endDate.getTime() - 90 * 24 * 60 * 60 * 1000);
const formatDate = (d: Date) =>
  d.toLocaleDateString('en-US', { month: 'short', day: 'numeric', year: 'numeric' });
const assessmentPeriod = `${formatDate(startDate)} – ${formatDate(endDate)} (90 Days)`;
```

The "assessment period" printed in the report header is computed at **render time**, not at **extraction time**. Running `generate-report.ts` twice on the same `assessment-results.json` three days apart produces reports that disagree on the period boundaries.

**Impact.** A customer re-running the report tool on archived findings will see different dates on the header. If the same `assessment-results.json` is ever regenerated into a PDF for a follow-up meeting, the date on that PDF does not match the extraction run.

**Correct behavior.** The extraction's own timestamp and the window it covered should be recorded as findings (e.g., a `DataCount` named "AssessmentPeriodStart" / "AssessmentPeriodEnd" emitted by the discovery or usage collector), and the assembler should read them. If the findings don't carry the period, the report should print "Assessment period: unknown" rather than compute one from render-time wall clock.

### Finding 4 — V11 validator passes on empty input

[validation.ts:830-842](apps/worker/src/normalize/validation.ts#L830-L842):

```ts
function checkV11_FieldCompletenessSuppression(data: ReportData): ValidationRule {
  const fc = data.dataQuality.fieldCompleteness;

  if (fc.length === 0) {
    // Empty is OK — means the assembler already suppressed it (Task 0.5)
    return {
      id: 'V11',
      name: 'Field completeness suppressed when empty',
      severity: 'warning',
      passed: true,
      message: 'OK — field completeness suppressed (no data).',
    };
  }
  // ...
```

The rule's semantic is "if the field-completeness table is empty, the assembler correctly suppressed it." That is a valid UI check. But the rule's code cannot tell the difference between:

- **(a)** the extractor produced no field-completeness data, and the assembler deliberately skipped the table, **versus**
- **(b)** the extractor _did_ run field-completeness but silently failed, and the findings never made it through.

In case (a) the rule is correctly reporting `passed: true`. In case (b) the rule is a blind accomplice. The validator has no way to disambiguate, which makes `passed: true` on empty input a defect regardless of intent.

**Correct behavior.** The rule should distinguish "suppressed because no findings of the expected type exist" (OK) from "suppressed because the findings that _should_ have produced this section are missing relative to the collector's declared coverage" (fail or warn). This requires a separate "collector coverage manifest" that lists what each collector _should_ produce, which the validator can cross-check.

### Finding 5 — Naming smell: "synthetic summary findings"

[assembler.ts:1871, 2799-2803](apps/worker/src/report/assembler.ts#L1871):

```ts
// V5-9: filter synthetic summary findings from template count
const tmplFindings = findings.filter(
  (f) =>
    (f.artifactType === 'QuoteTemplate' || f.artifactType === 'SBQQ__QuoteTemplate__c') &&
    !f.findingKey?.includes('unused_templates_summary') &&
    !f.artifactName?.includes('unused_templates_summary')
);
```

The code is **correct** — it filters out a summary-aggregate row (emitted by the templates collector with a `findingKey` ending in `unused_templates_summary`) to avoid double-counting it in the per-template count. But the word "synthetic" in the comment is dangerously misleading given the user's concern: it makes the reader think the code is handling fabricated data.

**Recommendation.** Rename the comment to `aggregate summary rows` or `collector rollup rows`. No code change.

## 8. Mitigation plan

The mitigation is in three phases: **immediate** (before anything ships again), **structural** (the fail-fast primitive), and **systemic** (the sweep that applies the primitive everywhere).

### Phase 1 — Immediate (before next CTO-visible delivery)

**M-1. Delete the quarantined shim and its artifacts.**

```bash

rm apps/worker/output/assessment-report-v2.1.pdf
```

Rationale: the customer needs the corrected 111-page report in place of the v2.1 PDF before anything else happens. See the disclosure template in the prior conversation.

### Phase 2 — Structural (the fail-fast primitive)

**M-4. Introduce `requireFinding()` and `optionalFinding()` helpers and ban bare `?? 0` coercion in the report path.**

Proposed API (to live in `apps/worker/src/report/require-finding.ts`):

```ts
/**
 * Retrieve a finding whose absence is a legitimate "no data" case.
 * Returns `null` if the finding is absent. Caller MUST handle `null`
 * explicitly — usually by setting a section to null (suppresses the
 * section) or by rendering an "Insufficient data" marker.
 *
 * NEVER use `?.countValue ?? 0` as a replacement for this — that is
 * the silent-fallback pattern this helper exists to eliminate.
 */
export function optionalFinding<T>(
  findings: readonly AssessmentFindingInput[],
  predicate: (f: AssessmentFindingInput) => boolean,
  label: string
): AssessmentFindingInput | null {
  const found = findings.find(predicate);
  if (!found) {
    // Add a diagnostic so the validator can detect absence patterns.
    getReportDiagnostics().push({
      kind: 'finding-absent',
      label,
      severity: 'info',
    });
  }
  return found ?? null;
}

/**
 * Retrieve a finding that is declared REQUIRED by the current
 * collector coverage manifest. Absence is a hard error that fails
 * the report assemble step and surfaces as a visible error banner.
 *
 * Use this for findings that the extractor's collector-coverage
 * contract says MUST exist for the report to be trustworthy.
 */
export function requireFinding(
  findings: readonly AssessmentFindingInput[],
  predicate: (f: AssessmentFindingInput) => boolean,
  label: string
): AssessmentFindingInput {
  const found = findings.find(predicate);
  if (!found) {
    throw new ReportAssembleError(
      `Required finding '${label}' is missing. ` +
        `The collector that produces it did not run or failed silently. ` +
        `Report cannot proceed without this finding — fix the extraction first.`,
      { label }
    );
  }
  return found;
}
```

And a companion helper for numeric fields:

```ts
/**
 * Read a numeric field from a finding. Returns `null` if the finding
 * or the field is absent. NEVER returns a zero default — the caller
 * must decide what absence means (suppress, warn, or require).
 */
export function numericField(
  finding: AssessmentFindingInput | null | undefined,
  field: 'countValue' | 'usageLevel' /* etc. */
): number | null {
  if (!finding) return null;
  const v = finding[field];
  if (v === null || v === undefined) return null;
  if (typeof v !== 'number') return null;
  return v;
}
```

Rationale: replaces the `?? 0` pattern with a three-way choice:

1. **Required** → throw → visible error → assembler stops, renderer renders an error banner
2. **Optional-explicit** → returns `null` → caller must suppress the section or render "Insufficient data"
3. **Optional-default** → _deleted_. No one gets to default silently. The word `??` becomes a code-review smell in the report path.

### Phase 3 — Systemic (the sweep)

**M-5. Migrate the 105 `?? 0` sites to the fail-fast primitives.**

Not a single-commit change. Proposed process:

1. Start by **classifying each of the 105 sites** into one of: (a) required — finding must exist, absence is a hard error, migrate to `requireFinding`; (b) optional-explicit — finding may legitimately not exist, migrate to `optionalFinding` + explicit `if (!x) return null` in the caller; (c) legitimately-zero — the `?? 0` is a defensive guard on a known-numeric field that could genuinely be zero. The legitimately-zero class must be rare and each case must be justified in a code comment next to the fallback.
2. **Land the classification as a spreadsheet / JSON catalog** at `apps/worker/src/report/require-finding.catalog.ts`, reviewed and signed off before any migration code lands. This is the list of findings the report depends on.
3. **Migrate in small batches**, ~10 sites per commit, with focused tests that run the assembler against (a) the full golden fixture, (b) a fixture with the required finding removed (should throw), and (c) a fixture with the optional finding removed (should suppress the section cleanly).
4. **Lock down the pattern with a lint rule:** ban `??\s*0` and `??\s*['"]` in `apps/worker/src/report/` via a custom ESLint rule or a grep in the CI lint step. Allowed only inside `require-finding.ts`.

Effort estimate: **~3 engineer-days** for the catalog + primitive + first batch; **~2-3 days** for the full sweep depending on how many sites are required vs optional. Total ~5-6 days of focused work, but shippable in small increments.

**M-6. Fix wall-clock leakage in `assessmentPeriod` (Finding 3).**

Two options:

- **Option A (cleanest):** The `usage` collector emits a `DataCount` finding named `AssessmentPeriodStart` / `AssessmentPeriodEnd` at extraction time, with the `countValue` being the epoch ms. The assembler reads these with `requireFinding`. If missing → hard error (no period → no trustworthy conversion metrics → the report must stop).
- **Option B (minimal):** The assembler reads `extractedAt` from the findings file metadata (if available on the wrapping envelope). If missing → prints `Assessment period: unavailable` as a sentinel. Never computes from `new Date()`.
