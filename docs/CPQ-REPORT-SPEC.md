## 1. Overview

### What the Report Is

The CPQ Assessment Report is a PDF document generated by the extraction worker that analyzes a customer's Salesforce CPQ configuration and produces a comprehensive migration readiness assessment for Revenue Cloud Advanced (RCA). It covers product catalog, pricing rules, quoting patterns, customizations, approvals, and data quality.

### Current State

**V6 shipped** — the report is the final internal quality pass before SI handoff. 19 bundle-capable matches confirmed against Salesforce UI.

## 2. Architecture Decisions

### System Architecture

```mermaid
graph LR
    SF["Salesforce Org<br/>(REST, Tooling, Bulk APIs)"] -->|SOQL queries| C["Collectors<br/>(12 domain collectors)"]
    C -->|AssessmentFindingInput[]| FV["Findings Validator<br/>(V1-V16: data consistency)"]
    FV -->|findings + warnings| A["Assembler<br/>(findings → ReportData)"]
    A -->|ReportData| RV["Report Validator<br/>(V17-V24: cross-section consistency)"]
    RV -->|validated ReportData| T["Template<br/>(ReportData → HTML)"]
    T -->|HTML| R["Renderer<br/>(HTML → PDF)"]
```

**Pipeline:** Collectors extract raw findings from Salesforce. FindingsValidator checks data consistency pre-assembly. Assembler transforms findings into ReportData. ReportConsistencyValidator checks cross-section consistency. Template renders HTML. Renderer produces PDF via Playwright.

### Key Design Principles

1. **Single Source of Truth (ReportCounts):** Every metric that appears in more than one section is computed exactly once and consumed by reference. No section-level function may independently count findings for metrics covered by ReportCounts. Enforcement: code review + lint-like CI check (grep for forbidden `findings.filter(...).length` patterns on covered metrics).

2. **Two-Layer Validation:** See Section 3 for full design.

3. **Explicit Source Fields:** Metrics derived from Salesforce fields (`IsActive`, `SBQQ__Active__c`) must be preserved through the finding pipeline — not inferred from metadata heuristics.

### ReportCounts Interface

All metrics that appear in more than one report section are centralized in `ReportCounts`, computed once at the top of `assembleReport()`:

```typescript
interface ReportCounts {
  // Products
  totalProducts: number;
  activeProducts: number; // From IsActive field, NOT proxy
  activeProductSource: 'IsActive' | 'inferred' | 'unknown';
  bundleProducts: number;
  productOptions: number;
  productFamilies: number;

  // Rules
  activePriceRules: number;
  totalPriceRules: number;
  activeProductRules: number;
  totalProductRules: number;

  // Usage (90-day scope only)
  totalQuotes: number;
  totalQuoteLines: number;
  activeUsers: number;
  activeUsersSource: 'UserAdoption' | 'UserBehavior' | 'unknown';

  // Discount schedules
  discountScheduleTotal: number;
  discountScheduleUnique: number;

  // Packages
  sbaaInstalled: boolean;
  sbaaVersionRaw: string | null;
  sbaaVersionDisplay: string; // "v232.2.0" | "Installed (version unknown)" | "Not installed"

  // Code & automation
  approvalRuleCount: number;
  flowCountActive: number;
  flowCountCpqRelated: number;
  validationRuleCount: number;
  apexClassCount: number;
  triggerCount: number;
}
```

### Canonical Metric Definitions Registry

Every metric that appears in the report has a formal definition. This is the single source of truth for what each number means.

| Metric                      | Source Finding Type              | Explicit Field Basis                                    | Time Scope | Fallback                                                                         | Display Label                             |
| --------------------------- | -------------------------------- | ------------------------------------------------------- | ---------- | -------------------------------------------------------------------------------- | ----------------------------------------- |
| **Active Products**         | `Product2` findings              | `IsActive=true` (collector-extracted)                   | All-time   | `usageLevel !== 'dormant'` (degraded: label as "Products Extracted")             | "Active Products (IsActive=true)"         |
| **Total Products**          | `Product2` findings              | None (count all)                                        | All-time   | Always available                                                                 | "Total Products"                          |
| **Bundle-capable Products** | `Product2` findings              | `SBQQ__ConfigurationType__c IN ('Allowed','Required')`  | All-time   | `not_extracted` if field unavailable                                             | "Bundle-capable Products"                 |
| **Product Options**         | `ProductOption` findings         | `SBQQ__ProductOption__c` record count                   | All-time   | `not_extracted` if extraction skipped                                            | "Product Options"                         |
| **Product Families**        | Derived from `Product2.Family`   | Distinct non-null Family values on active products      | All-time   | "(none)" excluded from count                                                     | "Product Families (with active products)" |
| **Active Price Rules**      | `PriceRule` findings             | `SBQQ__Active__c=true`                                  | All-time   | `usageLevel !== 'dormant'`                                                       | "Active Price Rules"                      |
| **Active Product Rules**    | `ProductRule` findings           | `SBQQ__Active__c=true`                                  | All-time   | `usageLevel !== 'dormant'`                                                       | "Active Product Rules"                    |
| **Total Quotes (90d)**      | `DataCount` finding              | `SBQQ__Quote__c WHERE CreatedDate >= 90d ago`           | 90-day     | `not_extracted` if skipped                                                       | "Quotes (90 Days)"                        |
| **Total Quote Lines (90d)** | `DataCount` finding              | `SBQQ__QuoteLine__c WHERE Quote.CreatedDate >= 90d ago` | 90-day     | `not_extracted` if skipped                                                       | "Quote Lines (90 Days)"                   |
| **Top Quoted Product %**    | `TopQuotedProduct` findings      | Numerator and denominator both 90-day scoped            | 90-day     | 0% if denominator is 0                                                           | "% of Quotes (90d)"                       |
| **Active Users (90d)**      | `UserAdoption` finding (primary) | `countValue` from usage collector                       | 90-day     | Sum of `UserBehavior` `countValue` (labeled "Estimated")                         | "Active Users (90d)"                      |
| **Approval Rules**          | `AdvancedApprovalRule` findings  | `sbaa__ApprovalRule__c` record count                    | All-time   | `not_applicable` if sbaa not installed                                           | "Approval Rules (sbaa)"                   |
| **sbaa Version**            | `InstalledPackage` finding       | `NamespacePrefix='sbaa'`                                | All-time   | Fallback chain: OrgFingerprint → CPQSettingValue → "Installed (version unknown)" | "Adv. Approvals Version"                  |
| **Active Flows**            | `Flow` findings                  | `FlowDefinitionView WHERE IsActive=true`                | All-time   | `not_extracted` if skipped                                                       | "Active Flows"                            |
| **Validation Rules**        | `ValidationRule` findings        | `ValidationRule WHERE Active=true`                      | All-time   | `not_extracted` if skipped                                                       | "Validation Rules"                        |
| **Apex Classes**            | `ApexClass` findings             | Class contains SBQQ reference or is in dependency graph | All-time   | `not_extracted` if skipped                                                       | "Apex Classes (CPQ-related)"              |
| **Triggers**                | `ApexTrigger` findings           | Trigger object is SBQQ-namespaced                       | All-time   | `not_extracted` if skipped                                                       | "Triggers (CPQ-related)"                  |

### Metric Precedence Rules

1. **Primary source** always preferred (explicit SOQL result, dedicated finding type)
2. **Fallback source** used only when primary absent; value tagged with lower confidence
3. **If both miss**, display "Not extracted" — never display 0 for a metric that was supposed to be extracted

### Metric State Model

Display logic must distinguish true zero from unknown/unavailable:

| Status           | Meaning                          | Display                         |
| ---------------- | -------------------------------- | ------------------------------- |
| `present`        | Value extracted and confirmed    | Show value with Confirmed badge |
| `zero`           | Query succeeded, count is zero   | Show "0 — Confirmed"            |
| `estimated`      | Derived from fallback source     | Show value with Estimated badge |
| `not_extracted`  | Extraction failed or was skipped | Show "Not extracted" (never 0)  |
| `not_applicable` | Package/feature not installed    | Show "N/A" or omit section      |

V4 implemented status tracking for `activeProducts` and `activeUsers`. Generalization to all metrics is a future backlog item.

## 3. Quality Framework

### Root Cause Patterns

The bugs identified across V2-V4 clustered into two categories:

1. **Cross-source inconsistency** — the same metric computed differently in different assembler functions. Example: active products showed 38 in one section and 179 in another because two functions used different counting methods.

2. **Scope mismatch in collectors** — counts used different time windows or object filters. Example: top quoted product percentage exceeded 100% because the numerator was all-time and the denominator was 90-day.

### Mitigation Strategies

1. **ReportCounts centralization** — every multi-section metric computed once, consumed by reference (see Section 2)
2. **Two-layer validation** — pre-assembly and post-assembly checks (see below)
3. **Collector-level explicit field extraction** — `IsActive` field preserved through pipeline, not inferred
4. **Golden-file regression testing** — frozen fixtures produce deterministic ReportData snapshots; CI diffs catch regressions
5. **Canonical metric definitions** — formal registry eliminates ambiguous counting (see Section 2)
6. **SOQL verification** — disputed counts verified against direct Salesforce queries

### Two-Layer Validator Design

Validation code lives at `apps/worker/src/normalize/validation.ts`.

#### Layer 1: FindingsValidator (pre-assembly, V1-V16)

Operates on raw `AssessmentFindingInput[]`. Catches:

- Impossible values (negative counts, dates in future)
- Source-scope mismatches (all-time data labeled as 90-day)
- Missing dependencies (sbaa package detected but zero approval rule findings)
- Percentage math (> 100% on any finding)
- Duplicate finding keys

#### Layer 2: ReportConsistencyValidator (post-assembly, V17-V24)

Operates on assembled `ReportData`. Catches cross-section inconsistencies:

| Rule | Check                                                                                                                    | Severity |
| ---- | ------------------------------------------------------------------------------------------------------------------------ | -------- |
| V17  | Any percentage metric > 100% in assembled report                                                                         | Error    |
| V18  | `metadata.lowVolumeWarning` active users matches `cpqAtAGlance` active users                                             | Error    |
| V19  | For each metric in [activeProducts, totalProducts, bundleProducts]: every rendered instance must equal `counts.{metric}` | Error    |
| V20  | Complexity rationale mentions "no product options" when `counts.productOptions > 0`                                      | Error    |
| V21  | `counts.sbaaInstalled === true` AND rendered sbaa version display equals "Not installed"                                 | Error    |
| V22  | Approval rules section says "not detected" when `counts.approvalRuleCount > 0`                                           | Error    |
| V23  | Every Top Quoted Product has `percentQuotes <= 100`                                                                      | Error    |
| V24  | Appendix D Product Catalog says "not available" when `counts.productOptions > 0`                                         | Error    |

Additional validators added in V5-V6:

| Rule | Check                                                                              | Severity |
| ---- | ---------------------------------------------------------------------------------- | -------- |
| V25  | Section 6.6.1 approval rule count must equal Section 8.3 count                     | Error    |
| V26  | Report branding matches configured brand name (config-driven, disabled by default) | Warning  |
| V27  | Flow count in Appendix A must equal At-a-Glance                                    | Error    |
| V28  | Percentage table metadata must include denominator basis                           | Warning  |
| V29  | Displayed count must have basis label where multiple definitions exist             | Warning  |

### Coverage Model

Appendix D coverage claims follow per-category specific checks:

| Category                   | "Full" Requires                                      | "Partial" When                                    |
| -------------------------- | ---------------------------------------------------- | ------------------------------------------------- |
| Product Catalog            | Product2 + ProductOption findings                    | Products exist but no options                     |
| Pricing & Rules            | PriceRule + DiscountSchedule + CustomScript findings | Missing any sub-category                          |
| Transactional Data         | Quote + QuoteLine + UserBehavior findings            | Quote data but no user behavior / modification    |
| Custom Fields & Validation | ValidationRule + FormulaField + FieldCompleteness    | Validation rules exist but field completeness N/A |
| Custom Code                | ApexClass + ApexTrigger + Flow findings              | Missing triggers or flows                         |
| Quote Templates            | QuoteTemplate + TemplateSection findings             | Templates exist but no section detail             |
| Advanced Approvals         | AdvancedApprovalRule + CustomAction findings         | Only one type extracted                           |
| User Behavior              | UserBehavior findings with non-inferred source       | Only inferred data                                |

## 5. Lessons Learned

These architectural insights from V3-V6 directly inform the V2.1 work:

1. **Centralize all multi-section metrics.** The root cause of most cross-section inconsistencies was independent counting in different assembler functions. ReportCounts eliminates this by computing every shared metric exactly once. Any new metric that appears in 2+ sections must be added to ReportCounts first.

2. **Distinguish true zero from not-extracted.** Displaying 0 for a metric that was never queried is the most dangerous bug class in assessment reports. The metric state model (`present` / `zero` / `estimated` / `not_extracted` / `not_applicable`) prevents silent misreporting.

3. **Collectors must preserve explicit field values.** Inferring `IsActive` from `usageLevel` or other proxies produces incorrect counts. Collector-level extraction of the actual field value, with FLS fallback handling, is the only reliable approach.

4. **Two-layer validation catches different bug classes.** Pre-assembly validation (FindingsValidator) catches data-level issues. Post-assembly validation (ReportConsistencyValidator) catches rendering contradictions. Both are necessary — neither is sufficient alone.

5. **Coverage claims must reflect actual extraction depth.** Generic threshold checks (e.g., "count > 5 = Full") produce overclaims. Per-category rules with specific artifact type requirements are needed.

6. **Scope must be explicit in both code and display.** The 117% top product percentage bug occurred because numerator (all-time) and denominator (90-day) used different scopes. Every metric must document and enforce its time scope.

7. **Label every count with its basis.** "179 products" is ambiguous when both total and active counts exist. "179 total products (176 active)" eliminates SI confusion. This applies to all metrics where multiple definitions exist (product counts, rule counts, template counts).

8. **Golden-file testing catches assembler regressions; live testing catches collector regressions.** These are complementary, not interchangeable. CI uses frozen fixtures; live verification uses actual org data.
